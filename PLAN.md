# Implementation Plan: template-frontend-react

## Context

### Goal
Boosha needs a standardized, reusable template for React frontend projects. Any new React frontend project (SPAs, dashboards, internal tools) should be bootstrapped from this template with a working CI/CD pipeline, Docker setup, pre-commit hooks, and security scanning ‚Äî out of the box.

### What this template must deliver
A **standalone repository** that someone can clone/use as a GitHub template to start a new React frontend project. The pipeline lives at `.github/workflows/ci.yml` (root level) and **must be verified green** before the template is considered done.

### Tech stack
| Tool | Role | Why |
| ---- | ---- | --- |
| **Vite** | Build tool / dev server | Fast HMR, native ESM, minimal config. De facto standard for React SPAs. |
| **React + TypeScript** | UI framework + type safety | Strict mode TypeScript for compile-time error catching. |
| **pnpm** | Package manager | Fast, strict (no phantom dependencies), disk-efficient. Parallel to uv in the Python template. |
| **Biome** | Linter + formatter | Single Rust-based tool replacing ESLint + Prettier. ~100x faster, one config file. Parallel to ruff in the Python template. |
| **Vitest + React Testing Library** | Testing | Native Vite integration, jsdom environment, component testing with user-centric queries. |
| **React Router** | Client-side routing | Declarative routing with 2‚Äì3 pages to demonstrate the pattern. |
| **CSS Modules** | Scoped styling | Zero runtime cost, built into Vite, no extra dependency. Scoped class names prevent style leaks. |
| **nginx** | Production serving (Docker) | Lightweight, battle-tested static file server. SPA fallback config for client-side routing. |

**Conventions:**
- **Separate `tests/` directory:** Tests live in `tests/` at the root, mirroring the `src/` structure. Consistent with the Python template's approach and makes it clear what is application code vs. test code.
- **Feature folders:** Pages and components are grouped by feature in `src/` (`pages/Items/`, `components/Header/`), each folder containing the component and its CSS module.
- **Barrel-free:** No `index.ts` re-exports in folders. Imports use explicit file paths (`import { Home } from './pages/Home/Home'`). Barrel files cause circular dependency issues and make tree-shaking harder to reason about.

  - `typecheck` ‚Äî `tsc --noEmit` (type checking without emitting)
  - `test` ‚Äî `vitest run` (single run)
  - `test:watch` ‚Äî `vitest` (watch mode)
  - `test:coverage` ‚Äî `vitest run --coverage` (with coverage report)
- **Dependencies:**

- **Dev dependencies:**
  - `typescript`

**Why `react-router` v7 (not `react-router-dom`):** Starting with v7, React Router merged all packages into a single `react-router` package. `react-router-dom` is a legacy import path. The v7 package exports `BrowserRouter`, `Link`, `Routes`, etc. directly.

**Note:** All dependencies (including testing and routing) are declared upfront even though they're not used until Steps 2‚Äì3. This avoids re-running `pnpm install` and regenerating the lockfile in later steps.

#### `biome.json` requirements
- **Formatter:** indent style `space`, indent width `2`, line width `100`
- **Linter:** recommended rules enabled. Specific rule groups:
  - `correctness` ‚Äî catches real bugs (e.g., `useExhaustiveDependencies` for hooks)
  - `suspicious` ‚Äî flags likely errors (e.g., `noExplicitAny`)
  - `style` ‚Äî consistency (e.g., `useConst`, `useTemplate`)
  - `complexity` ‚Äî simplification (e.g., `noForEach`, `useFlatMap`)
- **Organize imports:** enabled (replaces ESLint `import/order` + Prettier `import-sort`)
- **Files ignored:** `dist/`, `coverage/`, `node_modules/`
- **Override for tests:** disable `noExplicitAny` in `*.test.tsx` files (test utilities sometimes require `any`)

**Why Biome over ESLint + Prettier:** Single Rust-based tool replacing two JavaScript tools. ~100x faster lint + format in one pass. One config file (`biome.json`) instead of `.eslintrc.*` + `.prettierrc` + `eslint-config-prettier` conflict resolution. Parallel to ruff replacing black + flake8 + isort in the Python template.

#### `tsconfig.json` requirements
- **Target:** `ES2022` (modern browsers, supports top-level await, class fields)
- **Module:** `ESNext` with `bundler` module resolution (Vite handles bundling)
- **Strict mode:** `true` (enables all strict checks: `strictNullChecks`, `noImplicitAny`, etc.)
- **JSX:** `react-jsx` (automatic JSX runtime ‚Äî no `import React` needed)
- **Path aliases:** `"@/*": ["./src/*"]` for clean imports (`@/components/Header/Header` instead of `../../../components/Header/Header`)
- **Include:** `src/`, `tests/`
- **Compiler checks:** `noUnusedLocals`, `noUnusedParameters`, `noFallthroughCasesInSwitch`

#### `vite.config.ts` requirements
- **React plugin:** `@vitejs/plugin-react`
- **Path alias:** resolve `@/` to `./src/` (must match `tsconfig.json` paths)
- **Vitest config (inline):** Vitest reads from `vite.config.ts` by default ‚Äî no separate `vitest.config.ts` needed.
  - `environment: 'jsdom'` ‚Äî simulates browser DOM for component tests
  - `globals: true` ‚Äî exposes `describe`, `it`, `expect` without imports (matches Jest convention)
  - `setupFiles: './tests/setup.ts'` ‚Äî runs before each test file
  - `css.modules.classNameStrategy: 'non-scoped'` ‚Äî uses original class names in tests so assertions can target them
  - `coverage.provider: 'v8'` ‚Äî fast, built-in V8 coverage
  - `coverage.reporter: ['text', 'lcov']` ‚Äî terminal summary + lcov for CI artifacts
  - `coverage.include: ['src/**/*.{ts,tsx}']` ‚Äî measure app code only
  - `coverage.exclude: ['src/vite-env.d.ts', 'src/main.tsx']` ‚Äî skip type declarations and entry point
  - `coverage.thresholds.lines: 80` ‚Äî minimum 80% line coverage (matches Python template threshold)

#### `index.html` requirements
- Standard Vite entry HTML: `<!DOCTYPE html>`, charset, viewport meta, title
- Mount point: `<div id="root"></div>`
- Script: `<script type="module" src="/src/main.tsx"></script>`

#### `src/vite-env.d.ts`
- Triple-slash reference to `vite/client` ‚Äî provides types for Vite-specific imports (`.svg`, `.module.css`, `import.meta.env`)

#### `src/main.tsx` requirements
- `createRoot` from `react-dom/client`
- Render `<App />` wrapped in `<StrictMode>`
- Mount to `#root`

#### `src/App.tsx` requirements
- Minimal placeholder component: renders a heading (e.g., `<h1>Template Frontend React</h1>`)
- No routing yet ‚Äî that comes in Step 2
- Just enough for Vite to build and serve a page

### Step 4: Dockerfile

**What:** Create a production-ready, hardened Dockerfile with multi-stage build, plus an nginx config for SPA serving.

**Files:**
- `Dockerfile`
- `.dockerignore`
- `nginx.conf`

**Multi-stage build (2 stages):**

**Stage 1 ‚Äî `builder`:** Installs pnpm and builds the production bundle. The Node runtime, pnpm, `node_modules/`, and all build tooling stay in this stage only.

**Stage 2 ‚Äî `runtime`:** Copies only the built `dist/` directory and the nginx config from `builder`. No Node, no pnpm, no source code ‚Äî just static files and nginx.

**Requirements:**

1. **Digest-pinned base images** ‚Äî e.g., `node:22-slim@sha256:<digest>` for the builder, `nginx:stable-alpine@sha256:<digest>` for the runtime. The human-readable tag stays for readability, the `@sha256:` suffix is what Docker resolves. Dependabot (Step 10) with `package-ecosystem: docker` opens PRs when a new digest is available.

2. **pnpm installation in builder:**
   - Enable corepack: `corepack enable` (ships with Node 22, manages pnpm version from `package.json`'s `packageManager` field)
   - `corepack prepare --activate` ‚Äî installs the exact pnpm version declared in `package.json`
   - This avoids pinning a separate pnpm version in the Dockerfile ‚Äî the single source of truth is `package.json`

3. **Layer optimization:**
   - Copy `package.json` + `pnpm-lock.yaml` first and run `pnpm install --frozen-lockfile`
   - Then copy the rest of the source and run `pnpm build`
   - This way, dependency install is cached across builds when only source code changes

4. **No dev dependencies in production:** Dev dependencies are needed at build time (TypeScript, Vite, etc.), so `pnpm install --frozen-lockfile` installs everything in the builder stage. But the runtime stage only receives `dist/` ‚Äî no `node_modules/` at all. The boundary between stages is the security/size gate.

5. **Runtime environment:**
   - `nginx:stable-alpine` ‚Äî small image (~10MB), only serves static files
   - Copy `dist/` to `/usr/share/nginx/html/`
   - Copy `nginx.conf` to `/etc/nginx/conf.d/default.conf`

6. **Non-root user:**
   - nginx on Alpine runs as `nginx` user (UID 101) by default when configured properly
   - Adjust `nginx.conf` to not require root: use `listen 8080` (unprivileged port), write pid to `/tmp/nginx.pid`, logs to stdout/stderr
   - Switch to `USER nginx` before CMD
   - Satisfies Kubernetes `runAsNonRoot: true` policies

7. **HEALTHCHECK:**
   - `HEALTHCHECK --interval=30s --timeout=5s --start-period=5s --retries=3 CMD wget -qO- http://localhost:8080/health || exit 1`
   - Uses `wget` (available in Alpine) instead of `curl` (not installed by default in Alpine)
   - The `/health` endpoint is handled by nginx (see nginx.conf below)

8. **Expose port 8080** ‚Äî unprivileged port (not 80, which requires root)

#### `nginx.conf` requirements

```nginx
server {
    listen 8080;
    server_name _;
    root /usr/share/nginx/html;
    index index.html;

    # Health endpoint for Docker HEALTHCHECK and orchestrator probes
    location = /health {
        access_log off;
        return 200 '{"status":"healthy"}';
        add_header Content-Type application/json;
    }

    # SPA fallback: serve index.html for all routes that don't match a file
    location / {
        try_files $uri $uri/ /index.html;
    }

    # Cache static assets aggressively (Vite adds content hashes to filenames)
    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # Security headers
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
}
```

Key decisions:
- **SPA fallback** (`try_files $uri $uri/ /index.html`) ‚Äî required for client-side routing. Without this, direct navigation to `/items` returns a 404 from nginx because no `items` file exists on disk.
- **`/health` endpoint** ‚Äî returns `{"status":"healthy"}` with `application/json` content type. Consumed by Docker HEALTHCHECK, Kubernetes liveness/readiness probes, and the CI smoke test. `access_log off` prevents health check spam in logs.
- **Asset caching** ‚Äî Vite outputs files like `assets/index-abc123.js` with content hashes. These are safe to cache forever (`immutable`) because a code change produces a new hash. The `1y` expiry covers CDN/browser caches.
- **Security headers** ‚Äî basic hardening: no iframe embedding (clickjacking), no MIME sniffing, controlled referrer. These are a starting point ‚Äî downstream projects may add CSP, HSTS, etc.

#### `.dockerignore` requirements
- `node_modules/` ‚Äî never send to build context (pnpm install runs inside the builder)
- `.git/` ‚Äî no git history in context
- `dist/` ‚Äî prevent stale local builds from leaking into the image
- `coverage/` ‚Äî test artifacts
- `.env`, `.env.*` ‚Äî secrets
- `*.md` ‚Äî docs not needed in image
- `.vscode/`, `.idea/` ‚Äî editor configs

**Note on Alpine vs Debian-slim for nginx:** Alpine produces a significantly smaller runtime image (~10MB vs ~70MB). The trade-off is musl vs glibc, but for a static file server this is irrelevant ‚Äî nginx on Alpine is battle-tested and widely used in production.

### Step 5: Pre-commit Hooks

**What:** Create `.pre-commit-config.yaml` with all required hooks.

**Hooks to include:**

1. **Standard hooks** (`pre-commit/pre-commit-hooks`):
   - `trailing-whitespace`
   - `end-of-file-fixer`
   - `check-yaml`
   - `check-json`
   - `check-added-large-files` (max 500KB)
   - `check-merge-conflict`

2. **Biome** (local hook):
   - Hook id: `biome-check`
   - Type: `local` (runs `pnpm biome check --write --no-errors-on-unmatched --files-ignore-unknown=true`)
   - Stages: `pre-commit`
   - File types: `ts`, `tsx`, `js`, `jsx`, `json`, `css`
   - Runs lint + format + import organization in a single pass on staged files
   - `--write` auto-fixes what it can; `--no-errors-on-unmatched` prevents failures on non-applicable files; `--files-ignore-unknown=true` skips files Biome doesn't understand (e.g., `.yaml`)

   **Why `local` hook type:** Biome doesn't maintain an official pre-commit mirror repo (unlike ruff which has `astral-sh/ruff-pre-commit`). The `local` hook type runs a command from the project's own tooling, which means it uses the exact version pinned in `package.json` ‚Äî no version drift between CI and pre-commit.

3. **TypeScript type check** (local hook):
   - Hook id: `typecheck`
   - Type: `local`
   - Command: `pnpm tsc --noEmit`
   - Stages: `pre-commit`
   - `pass_filenames: false` ‚Äî `tsc` always checks the entire project (it cannot type-check individual files because types flow across modules)
   - File types: `ts`, `tsx`

4. **Secret scanning** (`gitleaks/gitleaks`):
   - Catches accidentally committed secrets (.env values, API keys, tokens)
   - Critical for client-facing repos

5. **Conventional Commits** (`compilerla/conventional-pre-commit`):
   - Stage: `commit-msg`
   - Enforce `--strict` with types: `feat, fix, docs, chore, refactor, test, ci, build`

**Dev dependency:** `pre-commit` is NOT a Node package ‚Äî it's a Python tool. Unlike the Python template where `pre-commit` is bundled in `pyproject.toml`, for a frontend project we have two options:

- **Option A (chosen):** Require system-level `pre-commit` (`brew install pre-commit` or `pip install pre-commit`). This is the standard approach in non-Python repos. The README documents this as a one-time setup step.
- **Option B (rejected):** Use `husky` + `lint-staged` (Node-native git hooks). This avoids the Python dependency but loses the pre-commit ecosystem (gitleaks, conventional-pre-commit) and requires reimplementing those checks. Since Boosha already uses pre-commit in the Python template, consistency across templates wins.

**Post-clone setup required:**
```bash
# Install pre-commit (if not already installed)
brew install pre-commit   # macOS
# or: pip install pre-commit

# Install git hooks
pre-commit install
pre-commit install --hook-type commit-msg
```

#### Verification
```bash

pre-commit install --hook-type commit-msg

# 2. Run all hooks against existing files
pre-commit run --all-files
# All hooks should pass. On first run, pre-commit downloads and installs
# each hook's environment ‚Äî this is normal and only happens once.

# 3. Verify Biome hook auto-fixes
# Create a poorly formatted file, stage it, commit ‚Äî Biome should fix it
echo 'const   x  =  1;' > /tmp/test-biome.ts
cp /tmp/test-biome.ts src/test-biome.ts

# Expected: passes
```

### Step 6: CI Pipeline (`.github/workflows/ci.yml`)

**What:** The core of the template. A GitHub Actions pipeline at the root level.

**Critical:** The workflow file must live at `.github/workflows/ci.yml` in the root of the repo. GitHub Actions only discovers workflows in this exact path ‚Äî files placed elsewhere are invisible to the runner.

**Files created:**
- `.github/workflows/ci.yml`
- `.trivyignore` ‚Äî time-boxed CVE exception allow-list (empty by default, with format documentation)

**Workflow-level configuration:**
- **Triggers:** push to `main`, pull requests against `main`. `paths-ignore` skips docs-only changes (`*.md`, `docs/**`, `LICENSE`, `CODEOWNERS`, PR template).
- **Environment variables** (change per project):
  - `NODE_VERSION` ‚Äî Node.js version to install (e.g., `22`)
  - `APP_NAME` ‚Äî Application name
  - `DOCKER_IMAGE_NAME` ‚Äî Docker image name
- **Permissions:** least-privilege default `contents: read` at workflow level; jobs override per-need (e.g., `security-events: write` for SARIF upload, `packages: write` for ghcr.io push).

**No `PNPM_VERSION` env var.** `pnpm/action-setup@v4` reads the pnpm version directly from `package.json`'s `"packageManager": "pnpm@10.33.0"` field. Specifying both a `version` input and `packageManager` in `package.json` causes the action to error with "Multiple versions of pnpm specified". Single source of truth is `package.json`.

**Pipeline structure ‚Äî 4 jobs:**

#### Job 1: `quality` (Code Quality & Testing)
- **Runs:** Always (push to main, PRs)
- **Steps:**
  1. Checkout code (`actions/checkout@v4`)
  2. Install pnpm (`pnpm/action-setup@v4`) ‚Äî no `version` input; reads from `package.json`'s `packageManager` field
  3. Set up Node.js (`actions/setup-node@v4`) ‚Äî uses `NODE_VERSION` env var, `cache: 'pnpm'` (auto-caches pnpm store for faster installs)
  4. Install dependencies ‚Äî `pnpm install --frozen-lockfile` (id: `install`). `--frozen-lockfile` ensures CI uses the exact lockfile ‚Äî fails if lockfile is out of sync with `package.json`, preventing non-deterministic installs.
  5. Format check ‚Äî `pnpm biome format .` (id: `format-check`). Checks formatting without writing. Exit code 1 if any file is not formatted.
  6. Lint + import organization ‚Äî `pnpm biome lint .` (id: `lint`). Separate from format so failures are clearly attributed.
  7. Type check ‚Äî `pnpm tsc --noEmit` (id: `typecheck`). Runs the full TypeScript compiler without emitting files.
  8. Tests with coverage ‚Äî `pnpm vitest run --coverage` (id: `tests`). Vitest's inline config in `vite.config.ts` enforces ‚â•80% line coverage via `coverage.thresholds.lines`. Exit code 1 if below threshold.
  9. Upload coverage artifact ‚Äî **only on GitHub** (`if: always() && !env.ACT`), uploads `coverage/lcov.info` for downstream tools (Codecov, SonarQube). `if-no-files-found: ignore`.

**Fail-fast cascade:** Each step after install has `if: steps.<previous>.outcome == 'success'`. This stops the pipeline at the first failure instead of running every check and producing noisy output. Order is intentional ‚Äî format is fastest, lint catches more, typecheck is slower, tests are slowest.

#### Job 2: `dockerfile-security` (Static Analysis)
- **Runs:** Always (including locally under `act`); **Needs:** `quality`
- **Permissions:** `contents: read`, `security-events: write`
- **Steps:**
  1. Checkout code
  2. Install Hadolint ‚Äî **direct CLI download** (`curl` from GitHub releases, pinned to `v2.12.0`). Docker-based actions (`hadolint/hadolint-action`) fail under `act` due to a local image pull bug where `act` builds the action's Docker image locally but then tries to `docker pull` it from a registry. Direct CLI invocation is consistent with how Trivy image scan is handled in Job 3.
  3. Hadolint (Dockerfile lint) ‚Äî runs the CLI twice: once with `--format sarif` to produce SARIF output, once with `--failure-threshold error` for human-readable CI logs. Only error-level findings fail the build; style/info/warning surface in SARIF but do not block.
  4. Upload Hadolint SARIF to GitHub Security (`category: hadolint`) ‚Äî guarded: `!cancelled() && !env.ACT && hashFiles('hadolint-results.sarif') != ''` (publishes on both success and failure so error-level findings reach the Security tab; `hashFiles()` avoids phantom upload failures when no SARIF was produced)
  5. Trivy config scan ‚Äî `aquasecurity/trivy-action@master`, `scan-type: config`, `severity: CRITICAL,HIGH`, `exit-code: '1'` (any CRITICAL/HIGH misconfig fails the step)
  6. Upload Trivy config SARIF to GitHub Security (`category: trivy-config`) ‚Äî same guard as Hadolint upload

**Deviation from the Python template's Job 2:** The Python template uses `hadolint/hadolint-action@v3.1.0` (a Docker-based action). This template uses a direct CLI invocation because Docker-based actions fail under `act`. The Trivy config scan step is identical.

#### Job 3: `build` (Docker Build & Image Security)
- **Runs:** Always (including under `act`); **Needs:** `[quality, dockerfile-security]`
- **Permissions:** `contents: read`, `security-events: write`
- **Steps:**
  1. Checkout code
  2. Set up Docker Buildx (`docker/setup-buildx-action@v3`)
  3. Build Docker image ‚Äî `docker/build-push-action@v6`:
     - `push: false`, `load: true` (loads into local daemon for smoke test / scan)
     - `tags: ${{ env.DOCKER_IMAGE_NAME }}:${{ github.sha }}`
     - GHA cache only on GitHub, disabled under act via `${{ !env.ACT && 'type=gha' || '' }}` ternary
  4. Trivy image scan ‚Äî **direct CLI invocation**, not `aquasecurity/trivy-action`. First install Trivy with `aquasecurity/setup-trivy@v0.2.6` (pinned to `v0.69.3`). Then a plain `run:` step executes `trivy image --format sarif --output trivy-image-results.sarif --severity CRITICAL,HIGH --ignore-unfixed --ignorefile .trivyignore --exit-code 1 "$IMAGE_REF"`. **Why not `aquasecurity/trivy-action`?** Same reason as Python template: when `format: sarif` is combined with `exit-code: '1'`, the action runs Trivy twice ‚Äî once for SARIF, once for the gate ‚Äî and the `trivyignores` input is only wired to the SARIF run. Calling the CLI ourselves is one invocation, deterministic, and reproducible locally.
  5. Upload Trivy image SARIF (`category: trivy-image`) ‚Äî same guard pattern as Job 2
  6. Container smoke test ‚Äî runs the built image, waits up to 30s for readiness, then:
     - `curl http://localhost:8080/health` and verify `"status":"healthy"` (tolerating whitespace)
     - `curl http://localhost:8080/` and verify the response contains `<div id="root">` (proves Vite's built `index.html` is being served)
     - `curl http://localhost:8080/items` and verify the response contains `<div id="root">` (proves SPA fallback is working ‚Äî nginx returns `index.html` for a client-side route, not a 404)
     - Dumps container logs on failure; always stops and removes the `smoke-test` container
  7. Save Docker image to tar ‚Äî **only on main and on GitHub** (`if: github.ref == 'refs/heads/main' && !env.ACT`). Exports the smoke-tested, scanned image.
  8. Upload Docker image artifact (`actions/upload-artifact@v4`) ‚Äî same condition; `compression-level: 0`, `retention-days: 1`
  9. Cleanup built image ‚Äî **act only** (`if: always() && env.ACT`). Removes the tagged image to avoid local tag accumulation.

**Smoke test differences from Python template:** The Python template tests API endpoints (POST/GET JSON round-trip). The frontend template tests static file serving: is `index.html` served at `/`? Is the health endpoint responding? Does SPA fallback work for client-side routes? There's no API to test ‚Äî nginx serves files, not JSON responses.

#### Job 4: `push` (Registry Push)
- **Runs:** Only on push to main (`if: github.ref == 'refs/heads/main'` at job level). Under `act`, the job starts but every step is skipped via per-step `if: ${{ !env.ACT }}` guards. `env.ACT` is **not** valid in a job-level `if` ‚Äî act's schema rejects it, and GitHub's docs restrict job-level `if` to the `github`, `needs`, `vars`, and `inputs` contexts.
- **Needs:** `build`
- **Permissions:** `contents: read`, `packages: write`
- **Steps:** (every step carries `if: ${{ !env.ACT }}`)
  1. Verify registry credentials ‚Äî prints whether `REGISTRY_USERNAME`, `REGISTRY_TOKEN`, and `REGISTRY_URL` are configured, and **hard-fails with `exit 1` and an `::error::` annotation** if either secret is missing.
  2. Download Docker image artifact (`actions/download-artifact@v4`)
  3. Load Docker image ‚Äî `docker load -i /tmp/image.tar`
  4. Log in to registry ‚Äî `docker/login-action@v3`. Empty `registry` input ‚Üí defaults to Docker Hub.
  5. Tag and push ‚Äî plain `docker tag` + `docker push`. Target ref computed at step level: `<REGISTRY_URL>/<image>:<sha>` when `REGISTRY_URL` is set, otherwise `<image>:<sha>`.
  6. Registry push summary ‚Äî on success, writes a markdown table (Image / Commit / Registry) to `GITHUB_STEP_SUMMARY`. Guarded by `success() && !env.ACT`.

**Identical to the Python template's Job 4.** The push job is image-agnostic ‚Äî it loads a tar, tags it, and pushes it. No language-specific logic.

#### General notes (same as Python template)

**`env.ACT` for local detection:** `act` automatically sets `env.ACT=true`. This is the documented and reliable way to detect local execution.

**Push exactly what was scanned:** The `build` job saves the image as a tar artifact after scanning and smoke-testing it. The `push` job loads and pushes those exact bytes ‚Äî no rebuild.

**Registry-agnostic design:** Same secret names and variable as the Python template: `REGISTRY_USERNAME`, `REGISTRY_TOKEN`, `REGISTRY_URL`. The README documents configuration for Docker Hub, ghcr.io, AWS ECR (OIDC), ACR, and Harbor/Nexus/Artifactory.

**`.trivyignore` ‚Äî time-boxed CVE exceptions:** Same approach as the Python template. When a CRITICAL/HIGH CVE in the nginx base image has no fix yet, add it with `exp:YYYY-MM-DD`. Remove when Dependabot opens the base image bump PR.

**Local runs require `GITHUB_TOKEN`:** Same as the Python template. Composite actions need it to fetch install scripts under `act`.

```bash
# one-time setup
echo "GITHUB_TOKEN=$(gh auth token)" > .secrets

# every run
act push --container-architecture linux/amd64 --secret-file .secrets
```

`.secrets` **must** be gitignored (listed in Step 1's `.gitignore` requirements).

#### What runs where

| | GitHub | `act` (local) |
| - | ------ | ------------- |
| Quality checks (format, lint, typecheck, tests) | Yes | Yes |
| Hadolint + Trivy config scan | Yes | Yes |
| Docker build + image scan + smoke test | Yes | Yes |
| SARIF uploads (Security tab) | Yes | Skipped |
| Coverage artifact upload | Yes | Skipped |
| Registry push | Yes (main only) | Skipped |

#### Verification (completed)
```bash
# All three local jobs verified green with act:
act -j quality --secret-file .secrets --container-architecture linux/amd64
act -j dockerfile-security --secret-file .secrets --container-architecture linux/amd64 --pull=false
act -j build --secret-file .secrets --container-architecture linux/amd64 --pull=false

# Note: --pull=false is needed for jobs 2 and 3 to prevent act from
# attempting to re-pull locally cached images.
```

**Pre-requisite:** The pipeline has been verified locally with `act` (Step 6). This step covers GitHub-specific setup only.

#### 7a. Enable GitHub Actions

Repo ‚Üí **Settings ‚Üí Actions ‚Üí General**:
- "Allow all actions and reusable workflows" (or allow the specific actions used in `ci.yml`).
- Workflow permissions ‚Üí **Read and write permissions**. The per-job `permissions:` blocks in `ci.yml` narrow this to least-privilege, but the workflow-level setting must be permissive enough to allow `security-events: write` (SARIF uploads) and `packages: write` (GHCR push).

2. Repo ‚Üí Settings ‚Üí Secrets and variables ‚Üí Actions:

| Type | Name | Value |
|------|------|-------|
| Secret | `REGISTRY_USERNAME` | Your GitHub username |
| Secret | `REGISTRY_TOKEN` | The Classic PAT from step 1 |
| Variable | `REGISTRY_URL` | `ghcr.io/<your-username>` |

The resulting image target will be `ghcr.io/<username>/<DOCKER_IMAGE_NAME>:<commit-sha>`.

#### 7c. Push branch and open a PR to main


This triggers jobs 1‚Äì3 (`quality`, `dockerfile-security`, `build`). The `push` job is gated on `github.ref == 'refs/heads/main'` and will be correctly skipped on the PR.

Expected: `quality`, `dockerfile-security`, `build` green; `push` skipped.

After the PR is green, merge it. The push to `main` triggers a full run including the `push` job:


Expected: all 4 jobs green. The image appears at github.com ‚Üí your profile ‚Üí Packages (or the repo's Packages sidebar).

### Step 8: Branch Protection on `main`

**What:** Configure GitHub branch protection rules on `main` so direct pushes are blocked and every change goes through a PR with green CI.

**When to apply:** After Step 7's first GitHub run has **completed green**. GitHub's "Require status checks to pass before merging" dropdown is populated from historical workflow runs ‚Äî a check that has never executed is not selectable by name. The correct order is always: push ‚Üí first run completes green ‚Üí configure protection and select the now-visible check names.

**Rules to configure** (Settings ‚Üí Branches ‚Üí Add branch protection rule ‚Üí Branch name pattern: `main`):

1. **Require a pull request before merging** ‚Äî no direct pushes to `main`.
2. **Require approvals** ‚Äî minimum 1 reviewer.
3. **Require status checks to pass before merging** ‚Äî select every PR-triggered job from `ci.yml`: `quality`, `dockerfile-security`, `build`. The `push` job runs only on main, so it will not appear in the dropdown.
4. **Require branches to be up to date before merging** ‚Äî PR must be synced with latest `main` before merge.
5. **Do not allow force pushes** ‚Äî never on `main`.
6. **Do not allow deletions** ‚Äî never on `main`.
7. **Require linear history** (recommended) ‚Äî forces squash or rebase merges; no merge commits.

**Applies to two places:**
- This template repo itself ‚Äî configure once as part of shipping the template.
- Every new repo created from this template ‚Äî GitHub does **not** carry branch protection rules across template copies. The README (Step 12) must document this as a mandatory post-bootstrap step.

#### Verification
```bash
# 1. Direct push to main must be rejected
git checkout main
git commit --allow-empty -m "test: direct push"
git push origin main
# Expected: remote: error: GH006: Protected branch update failed

# 2. Force-push to main must be rejected
git push --force origin main
# Expected: remote: error: GH003: Sorry, force-pushing to main is not allowed
```

### Step 9: Git Submodule for Internal Docs
